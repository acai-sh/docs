## What is it?

Acai.sh is tooling for **Feature Requirements** (i.e. Acceptance Criteria) written in plain language. 

Acai gives each requirement an ID (ACID) that is unique, stable, and readible. These ACIDs are then synced to a datastore. This makes it easy to query, track, annotiate and monitor progress. Your feature requirements are accessible via CLI, MCP, and a web dashboard.

 The result is a workflow that largely replaces your messy GitHub and Linear tickets. This makes it possible for you, your agents, and your product manager to **share implementation progress, bugs, acceptance coverage, test coverage, product changes, and more.**

It is open source, and works for any tech stack. It can be incrementally adopted into existing product (frontend, backend, CLI, SDK, API, microservice, monolith, etc.)

## Very Quick Start

**Prerequisites:**  
✅ You build and ship incrementally, feature-by-feature.  
✅ You can write a good spec. You set a high bar for quality. User happyness is your primary focus.  

**Simple Example**  
Here is a sample spec, implementation, and tests.
<CodeGroup>

```yaml features/block-user/FRD.yaml
feature:
  name: Block User
  descripton: Block other users so they can't view my profile.
  slug: BLOCK

components:
  BUTTON:
    name: The block button
    requirements:
      1: Renders a button that says says "BLOCK" on all user profiles
      1-1: If looking at my own profile, it does not render
      1-2: If the user is already blocked, it says "UNBLOCK"
      2: On click, opens the Block User Menu

  MENU:
    name: The Block User Menu
    requirements:
      1: Renders the target user's profile name and a confirmation message
      2: On click, blocks the user
      2-1: On block, navigates away from the blocked profile
```

```tsx src/profile.tsx
// BLOCK.BUTTON.1
export function BlockButton({ userId, isOwnProfile, isBlocked }) {
  // BLOCK.BUTTON.1-1
  if (isOwnProfile) return null 
  // BLOCK.BUTTON.1-2
  return <button>{isBlocked ? 'UNBLOCK' : 'BLOCK'}</button> 
}
```

```ts test/profile.test.ts
describe('BLOCK.MENU', () => {
  it('BLOCK.MENU.1: renders profile name and confirmation', () => {...})
  it('BLOCK.MENU.2: blocks user on click', () => {...})
  it('BLOCK.MENU.2-1: navigates away after block', () => {...})
})
```

</CodeGroup>

**Core Workflow:**
1. **Specify.**  
Define your next feature by writing key requirements in `FRD.yaml`.
2. **Sync.**  
`acai push` syncs your spec to the datastore and tracker
3. **Implement.**  
Your agents pull relevant ACIDs and reference them in code and in tests.


# Key Features

## Turn your spec into tickets

Write simple, plain language, easy-to-read requirements. Run `acai push` and they are synced to your acai dashboard.

```
$ acai push

🎉 Success!
Synced 2 feature specs (15 features unchanged)

12 requirements changed
- 10 new ACIDs added
- 2 ACIDs deprecated or replaced
```

## Turn your tickets into assignments

The acai database becomes your agent's preferred source of truth. They pull requirements to inform implementation and plan the next PR.

```json
$ acai status block-user --json --status TODO

{
  "feature": "Block User",
  "slug": "BLOCK",
  "results": [
    {
      "id": "BLOCK.MENU.2",
      "requirement": "On click, blocks the user",
      "status": "TODO"
    },
    {
      "id": "BLOCK.MENU.2-1",
      "requirement": "On block, navigates away from the blocked profile",
      "status": "TODO"
    }
  ]
}
```

# Why acai?
**Greppable, traceable**  
Every requirement in your spec gets an id, like `BLOCK.MENU.1`. LLM agents love these. They are easy to grep, and provide an anchor to pull in more context as-needed.

IDs also help with code review. It's extremely useful to see "why" the developer or agent did something... like what specific requirement were they trying to satisfy?

**Stable, durable**  
Structured spec docs create a dependable source of truth, and continuous project history. Acai makes sure your code referencs are up-to-date and relevant.

**Rapidly iterate**  
By maintaining a spec at the feature level (rather than product or function level) it's effortless to 1-shot prototypes, create throwaway implementations, and improve the spec itself before touching production code. The result is that you spend far less time prompting and wrangling, more time on high-value engineering and product thinking.

**Self documenting**  
The spec is a trusted source of truth for generating documentation. Acai makes it possible to evolve your product without the spec becoming another outdated artifact.

**This is no longer tedious**  
Spamming IDs all over the code is a bit too painful for a human engineer, but it is painless for an agent.
Now it's actually possible to maintain spec as source of truth, and to guage how compliant your code is.
